# Contrat API proposé

Base `/api`, JSON UTF-8, HTTPS hors développement. Erreurs : `{ "error": { "code": "...", "message": "..." } }`, sans détail interne. Choix proposé : cookie de session/JWT opaque côté navigateur, `HttpOnly`, `Secure` en HTTPS, `SameSite=Lax` ou `Strict`; jamais dans localStorage/sessionStorage. Les requêtes mutantes authentifiées exigent un jeton CSRF lié à la session (`X-CSRF-Token`) selon ADR 0003.

Les valeurs de politique restent à valider. Aucun endpoint métier n’est encore implémenté, sauf health.

## Règles transverses

- Pas de CORS permissif avec credentials; allowlist d’origines. Limites de taille, schémas stricts, UUID valides et contenu JSON obligatoire.
- Les réponses d’authentification ne distinguent jamais utilisateur inconnu, mauvais mot de passe, désactivation ou verrouillage. `internal_failure_reason` reste en base avec valeurs contrôlées.
- `password_hash`, traces, SQL, configuration, JWT, cookie et CSRF ne figurent jamais dans les réponses ou logs. Les exceptions sont corrélées et détaillées seulement dans un canal interne expurgé.
